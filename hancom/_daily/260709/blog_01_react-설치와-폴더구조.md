# 웹 개발 8일차 (2) — React 첫 설치, 우르르 생긴 폴더들 하나씩 뜯어보기

> HTML + CSS + JS만 만지다가 드디어 오늘 React를 처음 깔아봤다.
> 근데 명령어 한 줄 쳤더니 폴더랑 파일이 우르르 생겨서 "이게 다 뭐지…?" 싶었음.
> 그래서 이 글은 **"React 설치하면 나오는 것들, 대체 각자 뭐 하는 애들인지"** 하나씩 뜯어보는 글이다.

> 🖼️ **이미지 자리** — 글 대표 썸네일. `localhost:5173`에 처음 뜬 React 기본 화면(Vite + React 로고 도는 페이지) 스크린샷이면 딱 좋음.

---

## 0. 오늘의 요약

- React는 그냥 파일 하나 만든다고 되는 게 아니라 **Node.js → Vite** 를 거쳐서 프로젝트를 "생성"한다.
- 명령어 한 줄이면 폴더·파일·부품이 자동으로 좌르륵 깔린다.
- 그렇게 생긴 파일들이 각자 역할이 있는데, 그걸 오늘 하나씩 파악했다.
- 핵심 흐름은 딱 이거다 → **`index.html`(빈 액자) → `main.jsx`(다리) → `App.jsx`(진짜 화면)**

---

## 1. React 쓰려면 왜 Node.js부터 깔아야 함?

처음에 이해가 안 갔다. "React 배우는데 왜 갑자기 Node.js를 깔라는 거지?"

교재에서 든 비유가 딱 와닿았다.

> 완성된 장난감은 아이가 갖고 놀지만, **그 장난감을 찍어내는 공장 기계**가 없으면 안 된다.

- **Node.js** = 그 공장 기계. React 코드를 실제로 돌리고 빌드해주는 실행 환경.
- **npm** = Node.js 깔면 같이 딸려오는 **부품 설치 도구**. (`npm install ...` 하면 필요한 라이브러리를 가져다 준다)

즉 React 코드 자체는 내가 짜지만, 그걸 **돌려주고 찍어내주는 기계가 Node.js** 라는 것. 그래서 얘부터 깔아야 함.

> 🖼️ **이미지 자리** — 터미널에서 `node -v`, `npm -v` 쳐서 버전 뜨는 스크린샷. (설치 확인 느낌)

---

## 2. Vite로 프로젝트 생성 — 명령어 한 줄의 마법

옛날엔 React 프로젝트 하나 세팅하는 것도 일이었다는데, 요즘은 **Vite** 라는 도구가 명령어 한 줄로 다 해준다.

```bash
npm create vite@latest my-app
```

이거 한 줄 쳤더니… 폴더랑 파일이 **우르르** 생겼다. 이게 오늘 제일 당황스러웠던 부분ㅋㅋ

그리고 생성 후에는 이 순서로 실행한다:

```bash
cd my-app          # 만들어진 폴더로 이동
npm install        # 부품(node_modules) 설치
npm run dev        # 개발 서버 켜기 → localhost:5173
```

> 🖼️ **이미지 자리** — `npm create vite@latest` 실행하면 프레임워크 고르라고 나오는 선택 화면(React / Vue … 고르는 부분) 스크린샷.

### Vite가 뒤에서 해주는 일 (★ 중요)

Vite는 그냥 폴더만 만들어주는 게 아니라, 개발하는 내내 이 세 가지를 자동으로 해준다:

1. **JSX → 순수 JS 변환** — 내가 쓴 JSX(HTML처럼 생긴 문법)를 브라우저가 알아듣는 진짜 JS로 실시간 번역.
2. **미리보기 서버 운영** — `localhost:5173` 주소로 내 작업물을 바로 띄워줌.
3. **저장하면 즉시 반영** — 새로고침 안 눌러도, 코드 저장하는 순간 화면이 바뀜. (이거 진짜 편함)

> 한 줄 요약: **Vite = "내가 짠 React 코드를 브라우저 화면으로 실시간으로 띄워주는 애"**

## 3. 그래서 생긴 폴더/파일들 — 하나씩 해부

명령어 한 줄로 생긴 구조는 대충 이렇게 생겼다:

```
my-app/
├── index.html          ← 빈 액자
├── package.json        ← 프로젝트 설명서(어떤 부품 쓰는지)
├── node_modules/       ← 설치된 부품 창고 (건드릴 일 X)
├── public/             ← 그냥 그대로 나가는 정적 파일(favicon 등)
└── src/                ← ★ 내가 실제로 작업하는 공간
    ├── main.jsx        ← 다리 (진입점)
    ├── App.jsx         ← 진짜 화면 내용
    ├── index.css       ← 전역 스타일
    ├── App.css         ← App 전용 스타일
    └── assets/         ← 이미지 등
```

> 🖼️ **이미지 자리** — VS Code 탐색기에서 `my-app` 폴더 트리 펼친 스크린샷. (`index.html` / `src/main.jsx` / `src/App.jsx` 세 개에 형광펜/동그라미 표시하면 뒤 설명이랑 연결돼서 좋음)

이제 핵심 3인방을 뜯어보자.

### ① `index.html` — 아무것도 없는 "빈 액자"

```html
<body>
  <div id="root"></div>
  <script type="module" src="/src/main.jsx"></script>
</body>
```

처음 열어보고 놀랐다. **내용이 진짜 아무것도 없음.** `<div id="root"></div>` 딱 한 칸이랑 `main.jsx` 불러오는 스크립트 한 줄이 전부다.

이 `#root` 라는 빈 칸이 바로 **React가 화면을 그려 넣을 액자**다. HTML은 그냥 빈 액자만 걸어두고, 그림은 React가 채운다.

### ② `main.jsx` — 액자와 그림을 잇는 "다리"

```jsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import './index.css'
import App from './App.jsx'

createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

해석하면 이렇다:

- `document.getElementById('root')` → `index.html`의 그 빈 액자(`#root`)를 찾아서
- `.render(<App />)` → 그 자리에 `App` 컴포넌트를 그려 넣어라

즉 `main.jsx`는 **"빈 액자(index.html) ↔ 진짜 그림(App.jsx)"을 연결하는 다리** 역할. 진입점(entry point)이라고 부른다.

> `import './index.css'` 도 여기 있음 → 그래서 `index.css`가 **사이트 전역 스타일**이 되는 거였다. (밑에서 다시 설명)

### ③ `App.jsx` — 진짜로 화면에 보이는 내용

여기가 **매일 작업할 곳**이다. `localhost:5173`에서 눈에 보이는 모든 게 여기서 나온다. 앞으로 만들 부품(컴포넌트)들도 전부 여기서 조립한다.

> 🖼️ **이미지 자리** — 왼쪽엔 `App.jsx` 코드, 오른쪽엔 `localhost:5173` 화면을 나란히 둔 스크린샷. "이 코드가 → 이 화면" 이 한눈에 보이게.

## 4. 오늘 제일 헷갈렸던 것 — `index.css` vs `App.css`

CSS 파일이 두 개나 있어서 "이게 왜 나뉘어 있지?" 했다.

처음엔 나는 **"페이지가 달라서 나눈 건가?"** 하고 오해했다. 근데 지금 이 프로젝트는 페이지가 하나뿐이라 그 논리가 안 맞았음. 그래서 다시 정리한 진짜 기준은 이거다 👇

| 구분 | `index.css` | `App.css` |
|------|-------------|-----------|
| **누가 import 하나** | `main.jsx` | `App.jsx` |
| **대상** | `body`, `#root`, `h1`, `:root` 변수처럼 **전역·태그 단위** | `.hero`, `.ticks`처럼 **특정 클래스 전용** |
| **성격** | 사이트 전체 기본값(색, 폰트, 다크모드) | App 컴포넌트 안의 특정 요소 스타일 |

★ 정리하면 CSS가 여러 개로 나뉘어 있을 때 판단 기준은:

1. **누가 import 하는가**
2. **전역 기본값이냐 vs 특정 클래스 전용이냐**

"페이지가 다르다"가 아니었다. 앞으로 컴포넌트가 늘어나도 (전역 1개 + 컴포넌트마다 각자 css) 이 기준은 그대로 적용된다.

## 5. 구조 안 잡힐 때 팁 — "진입점부터 역추적"

새 프레임워크 구조가 막막할 때, 오늘 써먹은 방법:

> **빈 껍데기부터 거꾸로 따라가기**
> `index.html`(빈 div) → `main.jsx`(연결) → `App.jsx`(내용)

"빈 div 하나 두고, JS가 그 안을 채운다"는 패턴은 Vue든 뭐든 거의 똑같다고 한다. 그래서 처음 보는 프로젝트여도 **진입점(index.html)부터 역추적**하면 흐름이 빨리 잡힌다.

그리고 사실 이건 예전에 HTML/CSS/JS 파일 나눠 쓰던 거랑 원리가 같았다 — **구조(App.jsx) / 디자인(css) / 동작을 분리**하는 것. React는 여기에 **"컴포넌트 단위로 또 쪼갠다"** 는 개념이 하나 더 얹힌 것뿐. (이건 다음 글에서!)

## 마무리

React 설치라고 해서 파일 하나 뚝딱일 줄 알았는데, **Node.js 공장 → Vite로 생성 → 우르르 생긴 폴더** 라는 흐름이 있었고, 그 폴더들도 각자 역할이 딱 정해져 있었다.

핵심만 다시 붙잡고 가면:

- **index.html = 빈 액자**, **main.jsx = 다리**, **App.jsx = 진짜 화면**
- **CSS 나누는 기준 = 누가 import + 전역이냐 특정 클래스냐** (페이지 다름 ❌)
- 막히면 **진입점부터 역추적**

처음엔 폴더가 많아서 복잡해 보였는데, 하나씩 뜯어보니까 "아 얘는 이거 하는 애" 하고 이름표가 붙어서 은근 정리되는 느낌~ 다음 글에서는 드디어 이 `App.jsx` 안에서 **내 손으로 부품(컴포넌트)을 만들어 조립**한 실습기를 써보려 한다. 👉 [2편으로 이어짐]
