# 웹 개발 12일차 (3) — Vercel로 웹 배포 시작하기 (Drop & CLI)

> 지금까지 만든 건 전부 내 컴퓨터 `localhost`에서만 돌아갔다. 오늘은 드디어 **인터넷에 실제로 올려서 남한테 링크로 보여주는** 배포(deploy)를 배웠다.
> Vercel이라는 서비스를 썼는데, 처음엔 "배포 = 뭔가 엄청 복잡한 서버 설정"인 줄 알았는데 생각보다 훨씬 간단해서 좀 허무했다 ㅎㅎ. 가장 쉬운 드래그드롭부터, 실무에서 쓰는 CLI 배포까지 순서대로 정리했다.

---

## 0. 오늘의 요약

- **배포가 뭔지**: 내 로컬 파일(`index.html` 등)을 인터넷 서버에 올려서 누구나 접속할 수 있는 URL로 만드는 것.
- **Vercel Drop**: 파일을 브라우저에 드래그드롭만 하면 끝나는 "클릭 4번" 배포. 프로토타입엔 최고지만 **매번 새 프로젝트가 생겨서 기존 사이트 업데이트가 안 됨**.
- **Vercel CLI**: 터미널에서 `vercel` 명령으로 배포. `vercel`은 미리보기, `vercel --prod`는 실제 서비스 배포. **반드시 index.html이 있는 프로젝트 폴더 안에서 실행**해야 함.
- **삽질 포인트**: 처음에 엉뚱한 폴더에서 `vercel`을 쳐서 이상한 게 올라갔다. 루트 폴더 위치가 핵심.

---

## 1. 그래서 "배포"가 뭔데?

지금까지 나는 VS Code에서 파일을 만들고 Live Server나 `npm run dev`로 `localhost:5500`, `localhost:3000` 같은 주소로 확인했다. 그런데 이 주소는 **내 컴퓨터 안에서만** 열린다. 친구한테 `localhost:3000` 링크를 보내봤자 친구 컴퓨터엔 그 파일이 없으니 안 열린다.

**배포(deploy)**는 이 파일들을 인터넷 어딘가의 서버에 올려서, `https://내프로젝트.vercel.app` 같은 **진짜 URL**로 아무나 접속할 수 있게 만드는 것이다. Vercel은 이걸 무료로, 아주 쉽게 해주는 서비스다.

교재에서는 배포하는 방법을 크게 세 갈래로 소개했다:
1. **Drop** — 드래그드롭 (제일 쉬움, 이번 글)
2. **CLI** — 터미널 명령 (실무용, 이번 글)
3. **GitHub 연동** — git push하면 자동 배포 (다음 글에서)

---

## 2. 제일 빠른 방법: Vercel Drop (클릭 4번)

교재에서 "클릭 4번만에 배포"라고 소개한 방법이다. [vercel.com/new](https://vercel.com/new) 같은 Drop 페이지에 들어가서, 내 프로젝트 폴더를 **브라우저 창에 그냥 드래그해서 놓으면** 알아서 올라간다.

🖼️ **이미지 자리** — Vercel Drop 페이지에 프로젝트 폴더를 드래그드롭하는 화면 (직접 캡처해서 넣기)

정말 이게 끝이다. 코드 한 줄, 설정 한 개 없이 몇 초 만에 `.vercel.app` URL이 나온다. 프로토타입이나 "일단 이거 한번 보여줄게" 할 때는 이만한 게 없다.

**근데 결정적인 단점이 있다.** Drop으로 올리면 **올릴 때마다 매번 완전히 새로운 프로젝트가 생긴다.** 즉, 사이트를 고쳐서 다시 올리면 URL이 또 새로 생기고, 기존 URL은 그대로 옛날 버전으로 남는다. "이미 올린 사이트를 수정해서 업데이트" 하는 게 안 되는 것이다. 그래서 Drop은 **일회성 프로토타입 전용**이고, 계속 관리할 사이트는 아래 CLI나 GitHub 연동을 써야 한다.

---

## 3. 실무용: Vercel CLI로 배포하기

터미널에서 명령어로 배포하는 방법이다. 이게 진짜 개발자스러운 방법이고, 같은 프로젝트를 계속 업데이트할 수 있다.

### 3-1. Vercel 설치 & 로그인

먼저 Vercel CLI를 컴퓨터 전체에 설치한다.

```bash
npm i -g vercel
```

- `npm i` = `npm install`의 줄임말.
- `-g` = **global**. 이 프로젝트 폴더 안(`node_modules`)이 아니라 **컴퓨터 전체**에 설치하는 옵션. 그래야 어느 폴더에서든 `vercel` 명령을 쓸 수 있다.

설치가 끝나면 로그인한다.

```bash
vercel login
```

이걸 치면 이메일/GitHub 등으로 인증하는 절차가 뜬다. 한 번 로그인해두면 그 다음부턴 바로 배포할 수 있다.

### 3-2. 배포할 폴더로 이동 (이게 제일 중요)

**여기서 내가 삽질했다.** 배포는 반드시 **배포할 `index.html`이 들어있는 프로젝트 폴더 안에서** 실행해야 한다. 나는 처음에 상위 폴더에서 `vercel`을 쳐서, 엉뚱하게 여러 프로젝트가 든 폴더 전체가 잡혀버렸다.

```bash
# 배포할 index.html이 있는 폴더로 먼저 이동!
cd 07_groq/chatbot
```

Vercel은 "지금 터미널이 있는 위치(현재 폴더)"를 배포 대상으로 잡는다. 그래서 위치가 곧 배포 범위다.

### 3-3. 미리보기 배포: `vercel`

```bash
vercel
```

- 이걸 처음 실행하면 몇 가지 질문(프로젝트 이름, 폴더 확인 등)이 뜨고, 확인해주면 폴더에 **`.vercel`** 이라는 설정 폴더가 생긴다.
- 이 `.vercel` 폴더 덕분에 Vercel이 "이 폴더 = 이 프로젝트"라고 기억해서, 다음부터는 같은 프로젝트로 업데이트된다.
- `vercel`만 치면 **Preview(미리보기) 배포**가 된다. 실제 서비스 URL이 아니라, 테스트용 임시 주소가 나온다.

🖼️ **이미지 자리** — 터미널에서 `vercel` 실행 후 프로젝트 설정 질문 & Preview URL이 뜬 화면 (직접 캡처해서 넣기)

### 3-4. 실제 배포: `vercel --prod`

```bash
vercel --prod
```

- `--prod` = **production**. 진짜 서비스용 배포다.
- 이걸 실행하면 `https://프로젝트이름.vercel.app` 형태의 **정식 URL**로 사이트가 올라간다.
- 코드를 고친 뒤 다시 `vercel --prod`를 치면 **같은 URL이 새 버전으로 갱신**된다. (Drop과 달리 URL이 안 바뀜! 이게 핵심 차이다.)

🖼️ **이미지 자리** — `vercel --prod` 실행 후 Production URL과 "배포 완료" 메시지가 뜬 화면 (직접 캡처해서 넣기)

## 4. 정리 — Drop vs CLI

| 방법 | 명령/동작 | 장점 | 단점 |
|---|---|---|---|
| **Drop** | 브라우저에 드래그드롭 | 클릭 4번, 초간단 | 매번 새 프로젝트, 업데이트 불가 |
| **CLI (`vercel`)** | 터미널에서 명령 | 미리보기용 임시 URL | (테스트용) |
| **CLI (`vercel --prod`)** | 터미널에서 명령 | 같은 URL로 계속 업데이트 | 로그인·설치 필요 |

내가 이해한 핵심은 **"`.vercel` 폴더가 생기면 그 폴더가 하나의 고정된 프로젝트로 기억된다"** 는 것. 그래서 Drop과 달리 CLI 배포는 같은 사이트를 계속 관리할 수 있다.
