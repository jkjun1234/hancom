# 코딩 원칙 (Karpathy Guidelines)
출처: [multica-ai/andrej-karpathy-skills](https://github.com/multica-ai/andrej-karpathy-skills). Andrej Karpathy가 지적한 LLM의 흔한 코딩 실수(잘못된 가정을 검증 없이 진행, 과도한 복잡화, 무관한 코드 손대기)를 줄이기 위한 4가지 원칙. 아래 "모든 대화 규칙"(힌트 위주 튜터 모드)과 병행 적용하며, 실제로 코드를 작성·수정할 때(사용자가 이해했다고 하거나 직접 요청한 경우) 특히 따른다.

## 1. 코딩 전에 생각하기
- 가정을 넘겨짚지 말고 명시적으로 말한다. 불확실하면 진행하지 말고 묻는다.
- 해석이 여러 개면 조용히 하나를 고르지 말고 선택지를 제시한다.
- 더 단순한 방법이 있으면 말하고, 필요하면 사용자 요청에 이의를 제기한다.
- 애매한 부분이 있으면 멈추고 무엇이 애매한지 짚은 뒤 질문한다.

## 2. 최소 구현
- 요청한 것 이상은 만들지 않는다 — 불필요한 기능, 재사용 없는 코드의 추상화, 요청하지 않은 "유연성/설정 옵션"은 넣지 않는다.
- 일어날 수 없는 상황에 대한 에러 처리도 넣지 않는다.
- 200줄로 짤 걸 50줄로 줄일 수 있으면 줄인다. "시니어 개발자가 보면 과하다고 할까?"를 기준으로 판단한다.

## 3. 외과수술식 수정
- 기존 코드를 건드릴 땐 요청과 무관한 주변 코드·주석·포맷을 "개선"하지 않는다.
- 망가지지 않은 걸 리팩터링하지 않는다. 마음에 안 들어도 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견하면 지우지 말고 언급만 한다.
- 내 변경으로 인해 안 쓰게 된 import·변수·함수만 정리한다(원래 있던 죽은 코드는 요청 없이 지우지 않는다).
- 기준: 바뀐 줄 하나하나가 사용자 요청과 직접 연결되어야 한다.

## 4. 목표 기반 실행
- 작업을 검증 가능한 목표로 바꾼다(예: "버그 고쳐줘" → "재현하는 테스트를 먼저 작성하고 통과시키기").
- 여러 단계로 이루어진 작업은 `단계 → 검증` 형태로 짧게 계획을 먼저 밝힌다.
- 목표 기준이 뚜렷할수록 스스로 반복·검증하며 진행할 수 있다. "일단 되게 해줘" 같은 애매한 기준은 계속 되묻게 만든다.

**트레이드오프**: 이 원칙들은 속도보다 신중함 쪽으로 치우쳐 있다. 사소한 작업에는 판단해서 유연하게 적용한다.

## 나와 대화하며 항상 확인하고 기억해야할 파일

생성 위치는 hancom/_daily 폴더 내부 당일 날짜이름으로 된 폴더 내부(현재 날짜의 폴더이름이 존재하면 해당 폴더내에 파일을 생성)
에 날짜_Session_Backup 폴더(동일한 폴더명이 존재할 시 해당 폴더에서 진행)를 생성 후 파일을 생성하고 NN_대화주제명 으로 저장을 한다.
저장을 할땐 사용자가 세션저장해줘 라고 응답하여야만 저장해주고 말이 없이 세션 내용이 계속해서 길어지면 중간 저장할건지에 대해 추가 질문을 통해 저장여부를 추가 확인한다.

## 데일리 블로그 정리 규칙

### 발동 조건
- "오늘"과 "정리(블로그)"가 함께 담긴 요청(예: "오늘 내용 정리하자", "오늘 배운 거 블로그 정리해줘")이면 이 규칙을 실행한다.
- **단계마다 사용자에게 확인받으며** 진행한다(한 번에 자동 X).

### 0단계 — 날짜 확정
- 시스템 오늘 날짜를 기준으로 `YYMMDD`를 만들되, **"오늘 날짜가 YYMMDD 맞나요?"를 먼저 물어** 확정하고 진행한다.

### 1단계 — 자료 읽기(탐색)
- `hancom/_daily/YYMMDD/` 폴더를 확인한다. 없으면 만들고, 있으면 그 안에서 이어서 진행한다.
- 그날 노트 `hancom/_daily/YYMMDD/YYMMDD.md`를 읽고 `[NN]` 항목으로 그날 주제를 파악한다. **노트가 없으면 멈추고 사용자에게 알린다.**
- 폴더 안에 `blog_`가 아닌 중간 정리 노트가 있으면 함께 참고한다.
- 실습 코드를 찾는다. 주제마다 위치가 다르므로 **NN 폴더(예: `hancom/03_css/NN`, `hancom/04_js/NN`)와 단일 프로젝트(예: `hancom/05_react/my-app/src/components`) 모두** 추적한다. 무엇이 '오늘 것'인지 불확실하면 사용자에게 물어본다.
- **교재 사이트**를 WebFetch로 읽어 적극 참고한다: 일반은 `https://hancom-nine.vercel.app/index.html`, React는 `https://hancom-nine.vercel.app/react.html`. 그날 주제에 맞는 노드(NN)를 읽어 `[NN]` 항목과 매칭한다.

### 2단계 — 구성 제안(작성 전 확인)
- 주제 덩어리별로 몇 편(`blog_01`, `blog_02`…)으로 나눌지, 각 글의 **제목과 목차(섹션 구성)**를 제안하고 확인받는다.
- 제목의 'N일차' 번호는 기존 `blog_` 파일에서 추론해 +1로 제안하고 확인받는다.

### 3단계 — 이미지 계획(실행 전 질문)
- 초안에 `🖼️ **이미지 자리** — [구체 설명]` 형식으로 자리를 먼저 표기한다.
- 어떤 이미지가 이해에 도움될지 **후보를 제안**하고 사용자가 고르게 한다.
- 어떤 코드/화면을 **어떤 상태로 실행·캡처할지 실행 전에 물어본다.**
- 캡처 대상 구분:
  - 브라우저 렌더(localhost 등) → Playwright로 실제 실행 후 자동 캡처.
  - 터미널·VS Code 탐색기·플러그인 UI 등 자동화가 어려운 것 → 기본은 자리표시만 남기고 사용자가 직접 넣는다. **단, 실제 데이터(명령 출력·파일 구조·실제 렌더)를 HTML로 재현해 캡처할 수 있으면 그렇게 만들어 넣는다**(터미널 스타일·VS Code 탐색기 스타일·"코드+화면" 합성 등).
  - **재현 시 정직성**: 실제 출력을 못 구하는 것(예: 플러그인 분석 결과)은 '재현본'임을 사용자에게 밝히고, 그리기·표로 사실을 지어내지 않는다. 내용(구조·수치)은 실제 것만 쓴다.
- 버튼 클릭 등 상호작용이 필요한 화면은 **before/after 등 여러 장으로 세분화**한다(장수는 제안·확인).

### 4단계 — 글 작성
- **이미 작성된 초안이 있으면**(예: 강의장에서 미리 씀) 새로 쓰지 말고 그 글을 이어받아 진행한다 — (a) 코드 대조·보정, (b) 이미지 자리 채우기, (c) Velog 게시.
- 파일 경로: `hancom/_daily/YYMMDD/blog_NN_주제-슬러그.md`.
- 말투: **개인적·솔직한 TIL '~했다/함'**(260709 blog 스타일). 헤맨 점·오해했다 고친 것·구어체("ㅎㅎ")도 자연스럽게.
- 구조: 260709 blog 틀 그대로 — H1(N일차 부제) + 인용문 인트로 + `0. 오늘의 요약` + 번호 섹션 + 마무리(다음 글 연결).
- 코드: **실제 파일 코드와 똑같이** 싣는다 — 코드 블록을 실제 파일과 **한 줄씩 대조**해 다르면 실제에 맞춘다(특히 이어받은 초안은 필수. 예: 빠진 함수·줄). 단순화할 때도 "실제 코드엔 이런 게 더 있다"고 밝힌다. 초보자용 **라인/블록별 설명**을 붙인다.
- 깊이: 260709 blog 수준으로 상세히.

### 5단계 — 버그·미완성 코드 처리
- 실습 코드에 버그·미완성이 있으면: **버그났을 때의 실행 이미지/상황**과 **해결(정상) 상황의 실행 이미지**를 함께 보여주며, "왜 그랬는지 AI로 학습해 해결한" 흐름으로 서술한다.
- 내가 버그 해결을 시도했는데 안 되면, **무엇을 시도했고 왜 안 됐는지, 원인 추측**을 상세히 사용자에게도 알린다.
- 버그/정상 두 상태 이미지는 실제 파일을 **잠깐 수정 → 캡처 → 원복**하는 식으로 만든다(원본 보존. 예: Card의 버그 CSS로 캡처 후 `solid` 넣은 수정본으로 바꿔 캡처하고 되돌림).

### 6단계 — 이미지 캡처·삽입
- Playwright로 실제 실행(React는 `npm run dev`로 localhost, 정적 예제는 index.html)해서 캡처한다.
- React 캡처 실무:
  - `node_modules`가 없으면 **`npm install` 먼저**(설치 출력은 blog_01 설치 이미지로도 활용).
  - App.jsx에 대상 컴포넌트가 다 렌더돼 있지 않으면(주석처리 등), **임시 '캡처 하네스'**(App.jsx를 잠깐 교체해 대상 컴포넌트들을 한 화면에 렌더)로 캡처한다.
  - **캡처 후 임시로 수정한 원본 파일(App.jsx·CSS 등)은 반드시 원복**한다(미리 백업해두고).
- **타이트 크롭(여백 최소) + 2x 고해상도**(deviceScaleFactor 2, 텍스트 실제 bbox 기준으로 여백 최소화).
- 저장: `hancom/_daily/YYMMDD/images/`, 파일명 `blog_NN_설명_순번.png`(예: `blog_01_react-render_01.png`).
- md 삽입: **상대경로** `<img src="images/파일.png" width="...">`.

### 7단계 — Velog 임시글 게시
- **임시저장(초안)까지만.** 발행은 사용자가 직접 한다. (계정 `developjkj`)
- **로그인/세션**: Playwright 퍼시스턴트 프로필(`velog-profile`)로 로그인 세션을 유지한다. 세션이 없거나 만료되면 사용자가 직접 로그인해야 하며(크로미움에선 구글 OAuth가 막히니 **이메일 또는 GitHub 로그인**), 로그인 완료는 실제 인증 쿠키(`access_token`) 생성으로 확인한다.
- 이미지는 초안 에디터에 업로드해 velcdn URL로 교체한다(상대경로 → velcdn). **첫 업로드는 콜드스타트로 실패할 수 있으니 재시도 로직을 둔다.**
- **수정·보완은 새 초안을 또 만들지 말고 기존 `?id=<draftid>`를 열어 제자리 업데이트**한다. 각 글의 **draft id는 기록**해둔다(나중 수정용).
- 태그: 글 주제에 맞춰 자동으로 붙이고, 그날 글들을 하나의 **날짜 시리즈**로 묶을지 제안·확인한다.
- 업로드 과정에서 생긴 **내가 만든 중복·진단 초안만** id로 정확히 골라 정리한다. **사용자 본인 글이나 완성 초안은 절대 건드리지 않는다.**

### 뒷정리 — 작업 후 필수
- 임시로 수정한 실제 파일(App.jsx·CSS 등) **원복** 확인.
- 띄운 **dev 서버(vite 포트 등) 종료**, 만든 **임시 프로젝트 삭제**.
- 업로드 중 생긴 **내 중복·진단 초안 정리**(사용자 글은 절대 X).

### 8단계 — 최종 확인
- 글별 **제목 + 목차 + Velog 초안 URL + 삽입 이미지 수**를 표로 정리해 사용자에게 확인받는다.
- (제목·목차는 2단계 '작성 전'과 여기 '최종' 두 번 확인한다.)

### 9단계 — 깃(선택)
- 데일리 과제의 '깃 업데이트'는 **커밋 메시지를 제안하고 사용자 승인 후** 커밋한다(기본은 자동 커밋 안 함).
