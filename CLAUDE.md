## 추가 대화 규칙_1
모든 대화 내용세션은 답변해줌과 동시에 다른 세션에서도 완벽하게 흐름이 끊김없이 이어가도록 파일을 자동생성한다.
생성 위치는 hancom/_daily 폴더 내부 당일 날짜이름으로 된 폴더 내부(현재 날짜의 폴더이름이 존재하면 해당 폴더내에 파일을 생성)
에 날짜_Session_Backup 폴더(동일한 폴더명이 존재할 시 해당 폴더에서 진행)를 생성 후 파일을 생성하고 NN_대화주제명 으로 저장을 한다.
저장을 할땐 사용자가 세션저장해줘 라고 응답하여야만 저장해주고 말이 없이 세션 내용이 계속해서 길어지면 중간 저장할건지에 대해 추가 질문을 통해 저장여부를 추가 확인한다.

## 추가 대화 규칙_2

## 데일리 블로그 정리 규칙

### 발동 조건
- "오늘"과 "정리(블로그)"가 함께 담긴 요청(예: "오늘 내용 정리하자", "오늘 배운 거 블로그 정리해줘")이면 이 규칙을 실행한다.
- **단계마다 사용자에게 확인받으며** 진행한다(한 번에 자동 X).

### 0단계 — 날짜 확정
- 시스템 오늘 날짜를 기준으로 `YYMMDD`를 만들되, **"오늘 날짜가 YYMMDD 맞나요?"를 먼저 물어** 확정하고 진행한다.

### 1단계 — 자료 읽기(탐색)
- `hancom/_daily/YYMMDD/` 폴더를 확인한다. 없으면 만들고, 있으면 그 안에서 이어서 진행한다.
- 그날 노트 `hancom/_daily/YYMMDD/YYMMDD.md`를 읽고 `[NN]` 항목으로 그날 주제를 파악한다. **노트가 없으면 멈추고 사용자에게 알린다.**
- 폴더 안에 `blog_`가 아닌 중간 정리 노트가 있으면 함께 참고한다.
- 실습 코드를 찾는다. 주제마다 위치가 다르므로 **NN 폴더(예: `hancom/03_css/NN`, `hancom/04_js/NN`)와 단일 프로젝트(예: `hancom/05_react/my-app/src/components`) 모두** 추적한다. 무엇이 '오늘 것'인지 불확실하면 사용자에게 물어본다.
- **교재 사이트**를 WebFetch로 읽어 적극 참고한다: 일반은 `https://hancom-nine.vercel.app/index.html`, React는 `https://hancom-nine.vercel.app/react.html`. 그날 주제에 맞는 노드(NN)를 읽어 `[NN]` 항목과 매칭한다.

### 2단계 — 구성 제안(작성 전 확인)
- 주제 덩어리별로 몇 편(`blog_01`, `blog_02`…)으로 나눌지, 각 글의 **제목과 목차(섹션 구성)**를 제안하고 확인받는다.
- 제목의 'N일차' 번호는 기존 `blog_` 파일에서 추론해 +1로 제안하고 확인받는다.

### 3단계 — 이미지 계획(실행 전 질문)
- 초안에 `🖼️ **이미지 자리** — [구체 설명]` 형식으로 자리를 먼저 표기한다.
- 어떤 이미지가 이해에 도움될지 **후보를 제안**하고 사용자가 고르게 한다.
- 어떤 코드/화면을 **어떤 상태로 실행·캡처할지 실행 전에 물어본다.**
- 캡처 대상 구분:
  - 브라우저 렌더(localhost 등) → Playwright로 실제 실행 후 자동 캡처.
  - 터미널·VS Code 탐색기·플러그인 UI 등 자동화가 어려운 것 → 자리표시만 남기고 사용자가 직접 넣는다.
- 버튼 클릭 등 상호작용이 필요한 화면은 **before/after 등 여러 장으로 세분화**한다(장수는 제안·확인).

### 4단계 — 글 작성
- 파일 경로: `hancom/_daily/YYMMDD/blog_NN_주제-슬러그.md`.
- 말투: **개인적·솔직한 TIL '~했다/함'**(260709 blog 스타일). 헤맨 점·오해했다 고친 것·구어체("ㅋㅋ")도 자연스럽게.
- 구조: 260709 blog 틀 그대로 — H1(N일차 부제) + 인용문 인트로 + `0. 오늘의 요약` + 번호 섹션 + 마무리(다음 글 연결).
- 코드: **실제 파일 코드와 똑같이** 싣는다. 단순화할 때도 "실제 코드엔 이런 게 더 있다"고 밝힌다. 초보자용 **라인/블록별 설명**을 붙인다.
- 깊이: 260709 blog 수준으로 상세히.

### 5단계 — 버그·미완성 코드 처리
- 실습 코드에 버그·미완성이 있으면: **버그났을 때의 실행 이미지/상황**과 **해결(정상) 상황의 실행 이미지**를 함께 보여주며, "왜 그랬는지 AI로 학습해 해결한" 흐름으로 서술한다.
- 내가 버그 해결을 시도했는데 안 되면, **무엇을 시도했고 왜 안 됐는지, 원인 추측**을 상세히 사용자에게도 알린다.

### 6단계 — 이미지 캡처·삽입
- Playwright로 실제 실행(React는 `npm run dev`로 localhost, 정적 예제는 index.html)해서 캡처한다.
- **타이트 크롭(여백 최소) + 2x 고해상도.**
- 저장: `hancom/_daily/YYMMDD/images/`, 파일명 `blog_NN_설명_순번.png`(예: `blog_01_react-render_01.png`).
- md 삽입: **상대경로** `<img src="images/파일.png" width="...">`.

### 7단계 — Velog 임시글 게시
- **임시저장(초안)까지만.** 발행은 사용자가 직접 한다. (계정 `developjkj`)
- 이미지는 초안 에디터에 업로드해 velcdn URL로 교체한다(상대경로 → velcdn).
- 태그: 글 주제에 맞춰 자동으로 붙이고, 그날 글들을 하나의 **날짜 시리즈**로 묶을지 제안·확인한다.
- 업로드 과정에서 생긴 **내가 만든 중복·진단 초안만** id로 정확히 골라 정리한다. **사용자 본인 글이나 완성 초안은 절대 건드리지 않는다.**

### 8단계 — 최종 확인
- 글별 **제목 + 목차 + Velog 초안 URL + 삽입 이미지 수**를 표로 정리해 사용자에게 확인받는다.
- (제목·목차는 2단계 '작성 전'과 여기 '최종' 두 번 확인한다.)

### 9단계 — 깃(선택)
- 데일리 과제의 '깃 업데이트'는 **커밋 메시지를 제안하고 사용자 승인 후** 커밋한다(기본은 자동 커밋 안 함).
